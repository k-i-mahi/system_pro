\section{System Design Approach}
The system design follows a \emph{modular, layered} architecture. At the highest level the client interface is separated from API orchestration, domain services, persistence, and asynchronous workers. This organization reduces coupling among learning features (AI tutoring, routine scanning, community interactions) so each feature can evolve independently. All LLM calls are instrumented through a shared \texttt{withLogging} wrapper; all heavy inference tasks are queued through BullMQ workers so the HTTP thread stays responsive under load.

Ingestion runs as a BullMQ job so uploads never block the HTTP thread. Chunks are packed greedily up to 800 tokens with 150-token overlap; PDF chunks never cross page boundaries, which is what allows the answer layer to emit precise page-number citations. Embeddings are batched in groups of sixteen and written via raw SQL because Prisma cannot parameterise \texttt{vector(768)} columns.

For retrieval, BM25 top-20 and cosine top-20 run in parallel, then Reciprocal Rank Fusion with $K=60$ combines the ranks before selecting the top-8 context chunks. An optional cross-encoder re-rank stage is feature-flagged behind \texttt{ENABLE\_CROSS\_ENCODER} and disabled by default because BM25+cosine+RRF already achieves recall@5 above the 0.80 project target.

\section{Advanced OCR Sidecar}
\label{sec:ocr}
To keep the heavy Python machine-learning stack out of the Node.js backend, all advanced OCR runs in a separate FastAPI container. The backend's \texttt{extractTextFromFile} function accepts a \texttt{quality} parameter: \texttt{fast} runs Tesseract locally; \texttt{accurate} delegates to the sidecar via \texttt{undici}. A BullMQ \texttt{ocr.worker} guarantees slow inferences never starve request handlers.

\section{Agent ReAct Loop}
\label{sec:agent}
The chat controller delegates to a ReAct loop \cite{yao2023react} that alternates \emph{thought}, \emph{tool call}, and \emph{tool result} blocks before committing to a final answer. The loop is bounded at five iterations and twelve seconds wall-clock; every step is logged to \texttt{LlmCall} with a shared \texttt{parentCallId} so the instructor dashboard can reconstruct the full agent trace.

\noindent\textbf{Strategy selection rules} (highest-priority first):
\begin{enumerate}[label=\arabic*.,nosep]
    \item Detected confusion keyword (``I don't understand'', ``I'm lost'') $\Rightarrow$ \textsc{Socratic}
    \item Explicit worked-example request $\Rightarrow$ \textsc{Worked Example}
    \item Explicit explanation request $\Rightarrow$ \textsc{Explain}
    \item Posterior mean $< 0.4$ on active topic $\Rightarrow$ \textsc{Socratic}
    \item Posterior mean $> 0.75$ on active topic $\Rightarrow$ \textsc{Hint Ladder}
    \item Default $\Rightarrow$ \textsc{Explain}
\end{enumerate}

\section{Bayesian Mastery Model}
\label{sec:mastery}
Each (student, topic) pair carries a conjugate $\mathrm{Beta}(\alpha,\beta)$ posterior with uniform prior $\alpha=\beta=1$. Quiz submission updates the hyperparameters as:
\[
\alpha \leftarrow \alpha + N_{\text{correct}}, \qquad
\beta  \leftarrow \beta  + N_{\text{incorrect}}.
\]
The posterior mean is $\mathbb{E}[\theta] = \alpha/(\alpha+\beta)$; the 95\% credible interval uses the normal approximation:
\[
\mu \pm 1.96\sqrt{\frac{\alpha\beta}{(\alpha+\beta)^{2}(\alpha+\beta+1)}}.
\]
We adopt the uniform prior so a student who has answered nothing has mean $0.5$ and maximal uncertainty, which avoids the cold-start over-confidence visible in the previous $+0.1/-0.05$ rule. Adaptive difficulty targeting uses the posterior mean to select the next quiz difficulty: $d = \operatorname{clip}(\lceil 1 + 4\mu \rceil, 1, 5)$.

\section{Real-Time Notification Architecture}
Socket.IO is used for push notifications. The backend authenticates each Socket.IO connection with JWT, then places the socket into a room keyed by \texttt{user:\$\{userId\}}. When \texttt{createNotification} is called (from any service), it persists the row in the \texttt{Notification} table, then emits \texttt{notification:new} and \texttt{notification:count} events to the user's room. The frontend hook \texttt{useNotificationSocket} subscribes to these events and updates the TanStack Query cache in-place, so the bell badge and feed update instantly without polling.

\section{Methodology Summary}
The final methodology balances synchronous learning interactions (chat, RAG Q\&A, quiz) with asynchronous reliability features (ingestion worker, OCR worker, notification worker). Each sub-system has a deliberate degraded-mode out: fast-path OCR when the sidecar is unavailable; BM25-only retrieval when embeddings have not yet completed; and Tesseract on Nougat failure. This design trades some peak accuracy for predictable response times and high availability under CPU-only hardware constraints.
