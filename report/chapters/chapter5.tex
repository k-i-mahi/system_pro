\chapter{Societal, Health, Environment, Safety, Ethical, Legal and Cultural Issues}

\section{Introduction}
Software built for real classrooms touches questions that go beyond syntax and semicolons: who may see grades and attendance, how AI should be used without encouraging misconduct, and what happens when things break or data leaks. This chapter collects those considerations in the order suggested by the CSE~3200 report template, applied honestly to our implementation level.

\section{Intellectual Property Considerations}
The system is intended to host \textbf{teachers' and institutions' own materials}. Course notes, slides, and assessments remain the intellectual property of their authors. Our code does not transfer ownership of uploaded files to the platform authors; it stores and processes them only to provide the features described in this report. Third-party libraries, fonts, and frameworks are used under their respective open-source licenses, and any future public release would need a license audit beyond what this course project required.

If the product were deployed campus-wide, administrators would also need policies for \textbf{student-generated content} in threads and uploads, including retention and takedown rules.

\section{Ethical Considerations}
Put simply, the tool is meant to support learning, not to replace honest work on assignments or exams. Institutions should publish clear rules on when AI assistance is allowed. From a product stance, we avoided marketing language that promises automatic answers without human judgment.

Language models can reflect biases present in their training data. Course-grounded retrieval reduces some hallucination risk but does not remove fairness concerns. Tutors should review sensitive topics and intervene when automated explanations are inadequate.

\section{Safety Considerations}
For this type of web application, ``safety'' is largely about \textbf{safe failure modes}: authentication errors should not leak whether an email exists; destructive actions should require appropriate roles; file uploads should be validated to reduce malware and accidental huge payloads. Operational safety also includes backups and database migrations in a real deployment---only partially explored here.

\section{Legal Considerations}
A production deployment would need explicit attention to \textbf{data protection law} (consent, purpose limitation, export of personal data), \textbf{terms of use}, and potentially \textbf{child protection} rules if minors use the system. Our project documentation does not replace legal review; it simply records that these topics exist and would move to the top of the backlog before any real institution went live.

\section{Impact of the Project on Societal, Health, and Cultural Issues}
\textbf{Societal and cultural.} A unified academic workspace can reduce friction for students who already face cognitive overload. At the same time, it can concentrate power and visibility: analytics views should be designed proportionally so that ``engagement metrics'' do not unfairly label students.

\textbf{Health.} Long screen time and notification overload can affect wellbeing. The notification center is useful, but a mature product might add digests, quiet hours, and accessibility options so that the tool supports healthy study rhythms instead of constant interruption.

\section{Impact of the Project on the Environment and Sustainability}
Running local models and background workers consumes electricity and hardware lifetime. For a small academic deployment this is modest, but at scale, \textbf{energy use and cooling} matter. Sustainably, the more important point is \textbf{software longevity}: a modular, tested codebase is less likely to be thrown away and rewritten from scratch, which is a form of environmental impact even if it is hard to measure in kilowatt-hours.
