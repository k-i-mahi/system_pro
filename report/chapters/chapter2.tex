\chapter{Applications of the Work, Organization of the Project, and Related Works}

\section{Applications of the Work}
Cognitive Copilot is intended for \textbf{department-level academic use}: students can follow courses and topics, track routine and attendance, join classroom threads, receive notifications, and use the AI Tutor with material that belongs to the course. Tutors can manage class communication, review engagement through analytics, and guide students without sending them to a separate ``magic chat'' that lacks context.

The same design could be extended later toward stronger institutional deployment (identity federation, policy engines, richer grading exports). For this project, the emphasis is on a coherent demonstration system that behaves predictably for the stakeholders we modeled: students, tutors, and limited administrative control.

\section{Organization of the Project}
The repository is organized in a \textbf{monorepo-style} layout with a clear split between user interface, API, and infrastructure:
\begin{itemize}
\item \texttt{frontend/} --- React application, pages per module (courses, community, AI Tutor, analytics, and so on), shared libraries for API access and onboarding.
\item \texttt{backend-py/} --- FastAPI application structured into routers, services, schemas, and workers; PostgreSQL models and migrations live alongside the API code.
\item \texttt{docker-compose.yml} and related configuration --- bring up database, cache, API, workers, and optional AI runtime for end-to-end runs.
\item \texttt{report/} --- \LaTeX{} sources, figures, screenshots, and bibliography for this document.
\end{itemize}

This layout matches how we worked day to day: feature slices usually touched one router and its service on the backend, plus one or two pages on the frontend, with Docker used to validate integration.

\section{Related Works}

\subsection{Learning Management Platforms}
Conventional LMS platforms focus on course administration, assignment flow, and grading. Their strength is operational stability, but AI-assisted learning is often external or loosely connected. That creates a common gap: students can access information, but contextual tutoring is not integrated with the same course data.

\subsection{AI-Assisted Learning Systems}
Recent AI tutoring systems show the value of interactive question answering, guided explanation, and adaptive practice. However, many implementations rely on general chat with limited course grounding. In educational settings, answers are stronger when they are tied to real class materials and topic structure.

\subsection{Retrieval-Augmented Generation in Education}
Retrieval-Augmented Generation (RAG) combines information retrieval with language generation. In course settings, that helps because answers can be tied to available materials instead of relying only on model memory. Current research supports hybrid retrieval that blends lexical and vector signals \cite{lewis2020rag,gao2023ragsurvey}. In our project, RAG is used pragmatically to ground tutor answers in uploaded materials rather than to propose a new retrieval algorithm.

\subsection{OCR for Academic Workflows}
OCR has long been used for document digitization. Traditional OCR engines perform well on clean printed text, while transformer-based OCR is explored for harder cases. In educational workflows, OCR is mainly useful for extracting text from uploaded materials or timetable images so content can be indexed and reused.

\subsection{Agentic and Multi-Mode Tutor Interfaces}
A modern tutor interface often combines several patterns: free chat, explain mode, question answering, quiz practice, and resource recommendations. The literature on tool-using agents and structured prompting \cite{yao2023react,schick2023toolformer} informs this direction. For this report, the focus stays on implemented user-facing modes rather than speculative agent performance claims.

Based on the related context above, Cognitive Copilot is positioned as an \textbf{implementation-oriented} academic system that:
\begin{itemize}
\item integrates LMS modules and AI tutor workflows in one platform,
\item keeps AI interactions linked with course and topic context,
\item supports practical classroom operations (community, notifications, routine, analytics),
\item and prioritizes software correctness and maintainability.
\end{itemize}

The project contribution is therefore engineering integration and reliable module interoperability, rather than claiming a new AI algorithm.
