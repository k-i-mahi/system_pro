\section{Experimental Setup}
The project is implemented as a modular full-stack web application. The frontend is developed with React and TypeScript using Vite as the build tool. The backend uses FastAPI with a service-oriented structure and schema-based validation. PostgreSQL is the primary persistence layer, and Redis-backed asynchronous processing supports background operations and caching where configured.

Local runs use Docker Compose so that database, cache, API, workers, and optional AI services start with consistent ports and environment variables. This setup is what we used while capturing screenshots and running automated tests for the report.

\section{Evaluation Metrics}
Rather than reporting unsupported benchmark scores for the language model, we evaluated the engineering outcome with criteria that match the objectives in Chapter~I:
\begin{itemize}
\item \textbf{Correctness and safety of APIs:} route-level tests where available, plus manual checks for authorization on sensitive mutations.
\item \textbf{Coverage of core flows:} each bullet workflow in the list below can be exercised from the UI against a running stack.
\item \textbf{Build reproducibility:} backend test suite passes and frontend production build completes without errors on the reference environment.
\item \textbf{Interaction quality:} loading, empty, and error states behave sensibly on main pages; mutations show success or failure feedback.
\end{itemize}

\section{Dataset}
There is no single public ``course dataset'' bundled as a research corpus. Instead, the system is validated against \textbf{representative seeded or manually entered course data}: users, enrollments, topics, uploaded or pasted materials, routine slots, classroom threads, and notifications created during development. This is appropriate for a systems project: the scientific object is the platform behavior, not a fixed offline dataset. Where retrieval is enabled, chunks are derived from whatever materials tutors attach to topics in that environment.

\textbf{Authentication and profile.} Authentication uses token-based session flows. Protected endpoints rely on backend dependencies to enforce authorized access. Profile functionality includes user information updates, avatar handling, and account-level operations.

\textbf{Course and topic management.} Course management includes listing, detail views, enrollment-linked data, topic operations, and material actions. The backend and frontend are connected through dedicated route and service layers to keep CRUD behavior predictable.

\textbf{AI Tutor workspace.} The AI Tutor is implemented as a multi-mode workspace: chat, explain, ask-with-course context, quiz, and resource-oriented flows. Mode switching, topic context, and state synchronization are coordinated in the frontend and corresponding API endpoints.

\textbf{Routine and attendance.} Routine includes schedule slot management and calendar-style visualization in the UI. Attendance-related data feeds analytics and classroom views.

\textbf{Community and classroom operations.} Community features include classrooms, threads, replies, announcements, and role-dependent views for attendance and marks where implemented.

\textbf{Notifications.} Listing, unread counters, mark-read actions, and deep links into relevant pages are supported; the UI refreshes when new notifications arrive.

\textbf{Analytics.} Dashboard-style pages summarize activity for students and tutors, drawing on attendance and related entities as designed in the schema.

\subsection{Analysis of the results}
The implementation shows that a student team can deliver a \textbf{broad but coherent} LMS plus AI surface in one codebase when scope is controlled and integration is treated as a first-class requirement. The limiting factor is not drawing diagrams, but keeping authorization and validation aligned everywhere; that is why our evaluation narrative emphasizes tests and runnable deployment over model accuracy numbers we cannot fairly claim.

\textbf{Core flows exercised during validation.} The following project-core flows were checked repeatedly while preparing the demonstration:
\begin{enumerate}[label=\arabic*.]
	\item Secure access (login, protected routes).
	\item Course, topic, and material interactions.
	\item Routine and attendance interactions.
	\item AI Tutor modes with topic context.
	\item Community classroom, thread, and announcement flows.
	\item Notification list, unread state, and actions.
	\item Analytics page load for student and tutor perspectives.
	\item Settings and profile updates.
\end{enumerate}

\textbf{CRUD and permission mindset.} For each core entity we paid attention to create and update validation, consistent list and detail responses, delete behavior where exposed, and role boundaries on sensitive operations.

\section{Conclusion}
Taken together, the setup, metrics, screenshots, and objective mapping show a \textbf{working integrated system} rather than a disconnected set of demos. The most important lesson for us as developers is that AI features only feel trustworthy when ordinary LMS operations remain correct and fast. That balance---exciting tutor capabilities on top of boring, solid CRUD---is the result we want to highlight.
