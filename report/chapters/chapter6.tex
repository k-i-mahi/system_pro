\chapter{Addressing Complex Engineering Problems and Activities}

\section{Introduction}
Complex Engineering Problems (CEP) and Complex Engineering Activities (CEA) appear in accreditation-style documentation to show that a project demanded more than routine coding. For Cognitive Copilot, the complexity comes from \textbf{concurrency, security boundaries, heterogeneous components (browser, API, database, queue, AI runtime)}, and the need to keep a large feature surface consistent.

\section{Complex Engineering Problems Associated with the Current Project}
\begin{enumerate}[label=\arabic*.]
\item \textbf{Role consistency across the stack.} The same user may act as a student in one course and hold elevated privileges elsewhere. Keeping UI affordances, API authorization, and database constraints aligned is a distributed state problem, not a single \texttt{if} statement.

\item \textbf{Asynchronous ingestion and UI coherence.} When materials are processed offline, the system must expose status, avoid duplicate work, and refresh the UI without race conditions that show stale content.

\item \textbf{Grounded AI responses.} Connecting an LLM to live course data introduces failure modes: empty retrieval, partial chunks, model unavailability, and streaming errors. Handling those paths without breaking the rest of the page is an integration problem.

\item \textbf{Operational coupling in Docker.} Services depend on healthy startup order, environment variables, and persistent volumes. Diagnosing misconfiguration is part of the engineering problem, not an afterthought.

\item \textbf{Maintainability at project size.} As modules accumulate, duplicate validation logic and diverging DTO shapes become a long-term defect source. Controlling that drift is a design problem addressed through schemas, services, and tests.
\end{enumerate}

\section{Complex Engineering Activities Associated with the Current Project}
\begin{enumerate}[label=\arabic*.]
\item \textbf{Requirements and scoping.} Negotiating what belongs in ``core LMS'' versus optional stretch goals, and writing those boundaries into Chapters~I--IV of this report.

\item \textbf{Modeling and architecture.} Producing the diagrams in Chapter~III (use case, structural, behavioral, deployment, and data flow) to communicate structure to ourselves and to reviewers.

\item \textbf{Implementation and integration.} Building REST contracts, wiring the frontend query layer, and verifying cross-module flows such as notifications referencing classroom events.

\item \textbf{Verification.} Choosing meaningful automated tests and manual walkthroughs rather than only happy-path demos.

\item \textbf{Communication of limitations.} Documenting what is not enterprise-ready yet (Chapter~VII) is itself an engineering activity because it sets expectations for safe reuse.
\end{enumerate}

\section{Conclusion}
Mapping the work to CEP and CEA makes explicit what the repository already suggests: the value of Cognitive Copilot is in \textbf{how} the pieces were engineered together, not in any single isolated trick.
